// CURRENT PROBLEM

Two hard-coded driver columns (RUS_TIME / ANT_TIME) with a single shared DELTA, TIRE, and STATUS column. Breaks completely with 3+ drivers.

// OPTION A — SOLUTION

One column per driver. Each cell owns its own compound dot, lap time, and status tag. Delta is calculated as gap-to-fastest within each lap row. Scales to any number of drivers. Fastest cell in each row is highlighted. Session best gets purple accent. Personal best gets driver-color accent.

// CURRENT PROBLEM

Compound and Status are shared across all drivers in a row. A single SOFT label doesn't tell you who's on SOFT vs HARD. You lose all per-driver strategy context.

// OPTION B — SOLUTION

Each driver gets their own independent table section (band). Lap, time, delta-from-session-best, compound, status, and a delta bar are all per-driver. Stint boundaries are visible as dashed separators. Bands are collapsible. Adding a driver adds a new band — zero layout changes needed. Best for deep per-driver analysis.

// CURRENT PROBLEM

STATUS assumes all drivers had the same lap type (e.g., all FLYING). In reality drivers are on different stints, different compound ages, and different situations each lap.

// OPTION C — SOLUTION

Lap-centric rows: each row is a lap, each driver gets their own cell inside. Compound dot + time + status badge + delta are all per-driver per-cell. The fastest driver in each lap gets a subtle green highlight. No shared assumptions — a driver can be OUTLAP while another is FLYING. Good for lap-by-lap battle reading.

// ROOT_CAUSE_ANALYSIS — why the current design breaks
  • Delta column — Hard-coded as a single number assuming Driver A vs Driver B. With 3 drivers you need N−1 deltas or a shared reference. With 5 drivers the column becomes meaningless. Fix: make delta driver-local (gap from fastest in that lap, or from session P1).
  • TIRE column — Single column shared across the entire row implies all drivers are on the same compound for that lap. They are almost never on the same compound, especially across different teams or in different stint phases. Fix: put the compound indicator inside each driver's time cell.
  • STATUS column — Same issue. One driver can be on an OUTLAP while another is flying a hot lap. A single STATUS per row destroys this information. Fix: status is a badge per driver cell, shown only when non-FLYING (FLYING is implicit).
  • Column headers (RUS_TIME / ANT_TIME) — Baked driver codes into the schema. Adding a third driver means adding a new column — the header logic is not generic. Fix: driver list is dynamic; each driver is a column in the matrix or a band in the bands view.
// DELTA_STRATEGY — how to handle delta for N drivers
  • Option A (Matrix) — Delta = gap from fastest lap time recorded in that lap row across selected drivers. The fastest driver shows REF, others show +0.XXX. Only flying/valid laps count as the reference.
  • Option B (Bands) — Delta = gap from session overall best. Every driver's every lap is compared to the same absolute benchmark. Makes cross-driver analysis easy even when reading bands independently.
  • Option C (Lap Rows) — Delta = same as Option A (fastest in that lap), but visual emphasis is on who is fastest within each specific lap row.
  • Future: user-selectable reference — A toggle in the header could switch between "vs session P1", "vs fastest in lap", "vs personal best" — but this is a UI enhancement, not needed for the table layout decision.
// COMPOUND_STRATEGY — handling per-driver compounds
  • A small colored dot (6–7px) inline before the lap time carries all compound information without adding a column. Dot color: ● SOFT, ● MEDIUM, ● HARD, ● INTER, ● WET.
  • In Option B (Bands), a full COMPOUND column is shown because the band is single-driver — no ambiguity. The dot + abbreviation (S/M/H) is used for clarity.
  • Stint boundaries (compound changes) are shown as dashed row separators in both the matrix and bands views, making tire strategy visually scannable without a dedicated column.
// STATUS_STRATEGY — handling per-driver lap status
  • FLYING is the default state — show nothing. Reduce noise.
  • All other states get a small inline badge: OUT (outlap), IN (inlap), ABT (aborted/deleted), TRF (traffic), PRP (purple/session best).
  • PURPLE is the highest priority visual — a purple cell background + purple border + PRP badge. Hard to miss.
  • ABORTED laps show — for the time with a strikethrough-style indicator rather than a number, since the lap wasn't completed.

// RECOMMENDATION

  • Supplemental: Option B (Bands) — surface this as a toggle or alternate view mode when a single driver is selected, or when the user wants to inspect stint strategy.

// MOBILE PROBLEM

All three desktop options overflow a 375px screen. Side-by-side driver columns require ~600px minimum — cells become unreadable at mobile widths.

// MOBILE_A — DRIVER STACK

All of RUS's laps, then all of NOR's laps, etc. Each driver gets a sticky header with fastest + avg, followed by their full lap list. Stint boundaries shown as compound separators. Delta is vs session best. Full-width rows — no horizontal overflow. One scroll covers all drivers.

// LAYOUT_LOGIC
  • Sticky driver header — stays pinned while scrolling through that driver's laps. You always know whose data you're reading.
  • Stint separator — a compound-labeled divider appears whenever the tire changes. No TIRE column needed.
  • Delta = vs session best — every lap across every driver references the same 1:18.102 benchmark. Comparable across bands.
  • Scroll length — 4 drivers × 15 laps × 36px + headers ≈ 2,400px. Manageable. 30 laps ≈ 4,600px — consider a driver jump-nav at the top.
  • Status as badge, compound as dot — same encoding as desktop, no extra columns.
// TRADEOFF
  • Good for: focusing on one driver at a time, strategy review, single-driver mode.
  • Weaker for: direct lap-by-lap comparison. To compare RUS lap 7 vs NOR lap 7 you have to scroll between sections.
// MOBILE PROBLEM

Side-by-side driver columns can't compress to mobile widths without losing the data that makes them useful. Need a layout that reads well at one-column width.

// MOBILE_B — LAP BLOCKS

Each lap is a block. Inside each block, all drivers are stacked vertically — sorted fastest-to-slowest. Driver dot + code + compound + time + delta + status all on one row per driver. The fastest driver in each lap is always at the top. Lap-by-lap comparison is immediate — no scrolling between sections.

// MOBILE_B — LAP_BLOCKS (375px)
// 375px — IPHONE_SE
// LAYOUT_LOGIC
  • Fastest-first sort within each block — the session leader in that lap is always row 1. No scanning required to find the fastest.
  • Per-driver compound + status — each row owns its own dot and badge. VER on MEDIUM while others are on SOFT is immediately visible, side by side.
  • Delta = fastest in that lap (REF) — same as desktop Option A. The reference shifts every lap based on who's quickest.
  • Invalid laps (OUT/IN/ABT) pushed to bottom — flying laps always group at the top of each block for clean comparison.
  • Scroll length — 15 laps × (4 drivers × 36px + 32px header) ≈ 2,640px. ~5,200px for 30 laps.
// TRADEOFF
  • Good for: lap-by-lap battle reading, seeing who's fastest each lap, multi-driver comparison.
  • Weaker for: following one driver's full stint progression — they jump positions between blocks as the session evolves.
  • Complementary to Mobile_A — these two cover opposite reading directions. A toggle between them makes sense.
